## Description 

This transform is using [sentence encoder models](https://en.wikipedia.org/wiki/Sentence_embedding) to create embedding vectors of the text in each row of the input .parquet table.

The embeddings vectors generated by the transform are useful for tasks like document similarity, features extraction, etc which are also at the core of retrieval-augmented generation (RAG) applications.

The embeddings as well as the rest of the table columns can be stored as a [lancedb](https://lancedb.github.io/lancedb/) table or as a parquet file. If embeddings are first stored as parquet files, they can be converted to
a lancedb table afterwards.

### Input 

| input column name | data type | description |
|-|-|-|
| the one specified in _content_column_name_ configuration | string | the content used in this transform |


### Output columns


| output column name | data type | description |
|-|-|-|
| the one specified in _output_embeddings_column_name_ configuration | `array[float]` | the embeddings vectors of the content |
The output table is buffered by the transform worker until the total number of rows is large enough, if they are to be stored in lancedb. If not, there is no buffering and the output table is written out as a parquet file. If output is stored in lancedb, then the output table is written to lancedb as fragments. After the entire dataset is processed, the lancedb fragments are committed. A vector index can be built to allow for fast search of top-k documents whose embedding vectors are most similar to a given query embedding vector.

## Configuration

The transform can be tuned with the following parameters for creating embeddings and storing them either in parquet or in lancedb.
If you have already stored embeddings in parquet, you can also use this transform to convert them to lancedb.


| Parameter  | Default  | Description  |
|------------|----------|--------------|
| `model_name`                    | `"ibm-granite/granite-embedding-small-english-r2"` | The HF model to use for encoding the text. |
| `model_max_seq_length` | 2048 | The `max_seq_length` for the model and its `tokenizer.model_max_length`. |
| `content_column_name`           | `"contents"` | Name of the column containing the text to be encoded. |
| `output_embeddings_column_name` | `"embeddings"` | Column name to store the embeddings in the output table. |
| `lanceDB_data_uri`  | `"" ` | Path for lanceDB data. If stored in S3-like cloud object store, start with `"s3://..."` |
| `lanceDB_batch_size` | 524288 | The output table is buffered until it has at least 524288 records. |
| `lanceDB_fragments_json_folder` | `"" ` | A folder to store fragments related json info for the final `lance_commit` step. |
| `lanceDB_table_name` | `"" ` | The name of the lance table. |
| `embedding_batch_size` | 8 | The number of documents as a batch to create embeddings in a `model.encode()`. |
| `embeddings_exist` | False | Flag indicating if embeddings already exist in parquet files. |
| `embeddings_in_parquet` | False | Flag indicating if embeddings are to be stored in parquet files. | 
| `embeddings_in_chunks` | False | Flag indicating if each document is to be chunked first for chunk embeddings. |
| `embeddings_max_num_chunks` | 2 | If to be chunked first, the max num of chunks for each document. |

If the embeddings are to be stored in, or converted from parquet to, lancedb, a `lance_commit.py` needs to be executed to allow the
lancedb table to be available. To run `lance_commit.py`, the following paramters can be specified.

| Parameter | Default | Description |
|-----------|---------|-------------|
| `lanceDB_storage_type` | `"local"` | The storage type for lancedb, either `"local"` or `"s3"`. |
| `lanceDB_uri` | `""` | The lanceDB uri for setting up connection to the database. |
| `lanceDB_data_uri` | `""` | The lanceDB data path. |
| `lanceDB_table_name` | `""` | The name of the lanceDB table. |
| `lanceDB_fragments_json_folder` |`""`| The fragment json folder for `lance_commit.py`. |
| `lanceDB_table_schema_folder` | `""` | The folder containing the schema for the records stored in lanceDB. |


## Usage

### Launched Command Line Options 

When invoking the CLI, the parameters must be set as `--text_encoder_<name>`, e.g. `--text_encoder_lanceDB_data_uri "s3://my_project/lance/my_table.db/my_table.lance/"`. After embeddings are created and stored in a lancedb table, the parameters for `lance_commit` doesn't need the prefix, e.g., please see the `make run-python-cli-sample-lancedb:` in the [Makefile](Makefile)

## Running Text Encoder and Lance Commit via a Tekton Pipeline Example


Please see general [Tekton](../../tekton/README.md) examples that show a Tekton pipeline consisting of tasks that involve different transforms. A Tekton pipeline, which runs on a cluster in a cloud, can also be defined within the `text_encoder` transform. For example, once you login to a cluster, you can do `kubectl apply -f run-text-encoder-lance-commit.yaml` to launch a pipeline that is defined in `pipeline-text-encoder-lance-commit.yaml`, which starts with creating a repo, then launching a `text-encoder` ray job, and finally launching a `lance-commit` ray job. Details can be seen in these yaml files: [run-text-encoder-lance-commit.yaml](run-text-encoder-lance-commit.yaml), [pipeline-text-encoder-lance-commit.yaml](pipeline-text-encoder-lance-commit.yaml), [task-repo.yaml](task-repo.yaml), [task-text-encoder.yaml](task-text-encoder.yaml), [task-lance-commit.yaml](task-lance-commit.yaml), [text-encoder-rayjob.yaml](text-encoder-rayjob.yaml), and [lance-commit-rayjob.yaml](lance-commit-rayjob.yaml).

## Summary 
This project wraps the text_encoder transform with a Ray runtime. It creates embeddings, or taking existing embeddings already created and stored in parquet files, and stores the embeddings together with the other columns into a lancedb table. Alternatively, it can create and store embeddings in parquet files.
